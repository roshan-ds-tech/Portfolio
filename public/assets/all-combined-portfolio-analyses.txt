PORTFOLIO PROJECT ANALYSIS: 111 INTERNATIONAL SPA
======================================================

1. PROJECT OVERVIEW
-------------------
- Main Purpose: To serve as a digital storefront and premium showcase for "111 International Luxury Spa", presenting their high-end treatments, facilities, and testimonials.
- Real-World Problem: Luxury wellness centers require a digital presence that perfectly mirrors their physical experience. Standard, static websites often fail to convey the premium nature of the services, leading to poor brand perception and high bounce rates.
- Target Audience: High-net-worth clientele, wellness enthusiasts, and individuals seeking luxury relaxation, therapies, or family/couples spa experiences.
- Business Value: Acts as a primary lead generation and branding tool. It builds trust through high-end visuals and integrates an email newsletter subscription ("Inner Circle") to drive customer retention and repeat visits.

2. TECHNICAL ARCHITECTURE
-------------------------
- Frontend Architecture: Component-based Single Page Application (SPA) built with React 18 and Vite. It utilizes a custom layout wrapper (`PageLayout`) for seamless navigation between pages.
- Backend Architecture: Serverless/Static client-side application. No dedicated backend server is maintained.
- Database Architecture: An in-memory, read-only data layer. All 16 complex spa services are managed via a structured JavaScript object array (`src/data/spaServices.js`). 
- APIs & Integrations: Integrates with FormSubmit (`formsubmit.co`) via standard fetch API to handle newsletter form submissions without the overhead of a custom backend.
- Data Flow: Data is synchronously imported and queried on the client side using helper functions (e.g., `getServiceDetail`). State management is localized using React hooks (`useState`), ensuring zero-latency data hydration.

3. TECHNOLOGY STACK
-------------------
- Core Framework: React 18 with Vite (chosen for lightning-fast Hot Module Replacement and optimized production builds).
- Routing: React Router DOM v7 (chosen for declarative, client-side routing).
- Styling: Tailwind CSS, augmented with `class-variance-authority`, `clsx`, and `tailwind-merge` (chosen to build composable, highly reusable UI components akin to Shadcn UI patterns).
- Animation & Interaction: Framer Motion / Motion.js (chosen for complex reveal and scroll-linked animations) paired with `@studio-freight/lenis` (chosen to override browser scrolling physics for a buttery-smooth experience).
- Iconography: Lucide React (chosen for consistent, scalable SVG icons).

4. FEATURES ANALYSIS
--------------------
- Major Features:
  1. Immersive Homepage (Hero section, Core Values, Signature Rituals).
  2. Interactive Service Directory mapping 16 detailed treatments (Hammam, Hot Stone, 4-Hand Experience, etc.).
  3. Serverless Newsletter Subscription ("Inner Circle").
  4. Fluid Navigation Menu (`FluidMenu.jsx`) and Floating Contact components.
- Advanced/Hidden Features:
  - Zoom Parallax Gallery (`ZoomParallaxSection.jsx`): A scroll-linked animation component that dynamically scales images based on viewport scroll position.
  - Text Reveal Component (`TextReveal.jsx`): Custom stagger animations for typography on scroll intersection.
- Feature Ranking by Importance: 1. Service Directory 2. Immersive Homepage UX 3. Lead Capture/Newsletter.
- Feature Complexity: High complexity in UI/UX and animation synchronization; low complexity in data architecture.

5. ENGINEERING CHALLENGES
-------------------------
- Technical Challenges: Achieving a jank-free, 60fps experience while utilizing heavy CSS filters (backdrop-blur for glassmorphism) and complex DOM transforms.
- Implementation Details: Harmonizing Lenis smooth scroll with Framer Motion's `useScroll` and intersection observers required careful orchestrating to prevent layout thrashing.
- Architectural Decisions: Opting for a localized static data layer (`spaServices.js`) instead of a Headless CMS to guarantee 0ms latency page loads and simplify deployment.
- Tradeoffs: Used a third-party form handler (`formsubmit.co`) to save development time and infrastructure costs, trading off total control over the backend data pipeline.

6. DATABASE ANALYSIS
--------------------
- Design Patterns: Static Document Pattern. The `SPA_SERVICES` array acts as a NoSQL-like collection of documents.
- Relationships: Flat structure; components query the array via unique string IDs.
- Normalization: Denormalized design. Each service object self-contains all arrays for features, benefits, and image gallery paths, avoiding join logic.
- Data Integrity: Handled implicitly at development time since the data is strongly structured within the source code.

7. SECURITY ANALYSIS
--------------------
- Authentication/Authorization: Not applicable for this public-facing brochureware site.
- Validation: Implements basic HTML5 input validation for the email capture form.
- Security Best Practices: No sensitive data is stored on the client. The use of FormSubmit masks direct SMTP credentials from the frontend bundle.

8. PERFORMANCE ANALYSIS
-----------------------
- Optimization Techniques: Vite bundler handles JS minification and chunking. Tailwind CSS automatically purges unused styles. Images are loaded selectively.
- Efficient Algorithms: Fast `Array.prototype.find()` lookups scale perfectly given the small dataset size (16 items).
- Scalability Considerations: As a static SPA, the application is infinitely scalable via any edge CDN (Vercel, Netlify) without database connection bottlenecks.

PORTFOLIO ANALYSIS: AIR QUALITY & SMOKE LEAKAGE MONITORING SYSTEM
================================================================================

1. PROJECT OVERVIEW
--------------------------------------------------
- Main Purpose: A comprehensive IoT and machine learning-based system designed to collect real-time sensor data, classify current air quality metrics, and forecast future gas/smoke leakage risks.
- Real-World Problem Solved: Traditional smoke detectors are reactive, sounding alarms only after danger is present. This system proactively monitors air quality trends and uses predictive modeling to identify incoming hazards (like slow gas leaks or building smoke) before they reach critical levels.
- Target Users: Homeowners for smart home safety, industrial facility managers monitoring hazardous environments, and environmental researchers.
- Business Value: Provides an enterprise-grade, predictive safety mechanism that minimizes health risks and property damage. The multi-platform nature (Web, Mobile, R-Analytics) makes it a versatile product offering for IoT safety markets.

2. TECHNICAL ARCHITECTURE
--------------------------------------------------
- Complete Project Structure: A highly distributed architecture encompassing hardware (IoT), a centralized AI-powered backend API, a relational database, and four distinct frontend consumers (React, Flutter, Streamlit, R-Shiny).
- Frontend Architecture:
  - Web: React + Vite for a modern, responsive user dashboard.
  - Visualization: Streamlit for interactive machine learning insights and real-time inference alerts.
  - Mobile: Flutter for cross-platform iOS/Android monitoring.
  - Statistical: R-Shiny for enterprise-level statistical plotting.
- Backend Architecture: Flask REST API that ingests HTTP GET requests from the IoT hardware, processes data in-memory through pre-loaded AI models (Scikit-Learn RF and TensorFlow LSTM), and exposes data endpoints (`/latest`, `/history`) for frontend consumption.
- Database Architecture: SQLite managed via SQLAlchemy ORM, storing atomic sensor readings and their corresponding AI classifications.
- IoT/Hardware: ESP8266 (NodeMCU) reading analog data from an MQ135 gas sensor and transmitting it via WiFi.
- Data Flow: ESP8266 Sensor -> WiFi -> Flask Backend -> In-Memory Buffer (Deque) & ML Inference -> SQLite Database -> REST API -> React/Flutter/Streamlit/R-Shiny Clients.

3. TECHNOLOGY STACK
--------------------------------------------------
- Languages: Python, C++, JavaScript, Dart, R
- Backend: Flask, FastAPI (alternative API), SQLAlchemy
- Machine Learning: Scikit-Learn (Random Forest), TensorFlow/Keras (LSTM), Pandas, NumPy, Joblib
- Frontend: React (Vite), Streamlit, Flutter, R-Shiny
- Hardware: ESP8266, MQ135 Gas Sensor, Arduino IDE
- Why chosen: 
  - Python/Flask ensures rapid integration with ML/DL libraries.
  - TensorFlow LSTM is ideal for temporal time-series forecasting.
  - Flutter allows writing one codebase for both iOS and Android.
  - Streamlit and R-Shiny provide rapid, interactive data science visualization without writing complex frontend logic.

4. FEATURES ANALYSIS
--------------------------------------------------
- Major Features: 
  1. Real-time IoT data ingestion.
  2. Multi-output classification (Air Quality, Smoke Level, Gas Leakage, Risk Level, Pollution Trend).
  3. Time-series forecasting for predictive danger alerts.
  4. Multi-client real-time synchronization.
- Hidden/Advanced Features: 
  - A 10-minute sliding window mechanism using Python's `collections.deque` to feed dynamic 300-step sequences into the LSTM model for continuous prediction.
- Feature Ranking by Importance: 
  1. LSTM Predictive Forecasting (Highest complexity and value)
  2. IoT Sensor Integration
  3. Real-Time Multi-output Classification
  4. Cross-Platform Dashboards
- Complexity: Extremely high. It bridges hardware engineering, backend microservices, traditional ML, deep learning, and multi-platform frontend development.

5. ENGINEERING CHALLENGES
--------------------------------------------------
- Technical Challenge 1: Performing real-time deep learning inference on continuous time-series data. 
  - Solution: Implemented a highly efficient sliding window using `collections.deque(maxlen=300)`. This provides O(1) time complexity for appending new IoT readings and automatically evicts stale data, ensuring the LSTM always receives exactly 10 minutes of historical context without memory leaks.
- Technical Challenge 2: Cross-platform real-time data synchronization.
  - Solution: Centralized data logging into an SQLite database via SQLAlchemy, exposing lightweight JSON endpoints (`/latest`, `/history`) that React, Flutter, and R-Shiny clients can poll without overloading the ML inference engine.
- Architectural Decision: Loading AI models globally in memory upon server boot using `@st.cache_resource` and global variables in Flask. This prevents the high latency overhead of loading `.pkl` and `.h5` files on every single IoT sensor ping.

6. DATABASE ANALYSIS
--------------------------------------------------
- Design Patterns: Object-Relational Mapping (ORM) using SQLAlchemy.
- Tables/Relationships: A single, robust `sensor_readings` table designed for high-frequency time-series inserts.
- Data Integrity: Handled via SQLAlchemy models (`Float`, `DateTime`, `Boolean`) ensuring that the raw float values from the ESP8266 and the string classifications from the Random Forest model are strictly typed.

7. SECURITY ANALYSIS
--------------------------------------------------
- Current State: Operates primarily on local network trust (LAN).
- Security Implementations: Cross-Origin Resource Sharing (CORS) is configured to allow frontend communication. 
- Areas for Improvement: Missing JWT authentication for API endpoints, and HTTPS/SSL for data encryption in transit between the IoT device and the server.

8. PERFORMANCE ANALYSIS
--------------------------------------------------
- Optimization Techniques: Use of `deque` for memory-safe rolling windows. Model pre-loading in global scope.
- Scalability Considerations: SQLite is suitable for a single-node IoT setup. For industrial scaling with thousands of sensors, the architecture would need to transition to a time-series database (like InfluxDB) and a message broker (like MQTT or Kafka) instead of synchronous HTTP GET requests.

1. PROJECT OVERVIEW
- Main Purpose: A full-stack data intelligence dashboard designed to analyze, visualize, and forecast global COVID-19 epidemiological data.
- Real-World Problem: Public health data is often scattered, raw, and difficult to interpret. This project transforms massive, raw time-series data of confirmed cases and deaths into an interactive, visually digestible format while providing predictive insights into future spread.
- Target Audience: Epidemiologists, data analysts, public health officials, researchers, and the general public seeking to understand pandemic trends and geographic distribution.
- Business Value: Serves as an enterprise-grade data intelligence platform that provides actionable insights, real-time geographic tracking, and AI-driven forecasting, enabling data-driven decision-making.

2. TECHNICAL ARCHITECTURE
- Complete Structure: A containerized microservices architecture utilizing Docker Compose to orchestrate a Next.js frontend, a FastAPI backend, and a supplementary Streamlit prototyping layer.
- Frontend Architecture: Built with Next.js and React. Employs a modern, responsive layout styled with Tailwind CSS. Utilizes `react-plotly.js` for complex, interactive data visualizations. State management handles dynamic date and country filtering.
- Backend Architecture: A high-performance Python FastAPI REST server. It loads pre-processed data into memory at startup for low-latency querying and uses Pandas for dynamic data aggregation and Scikit-learn for on-the-fly predictive modeling.
- Database Architecture: Utilizes a file-based analytical datastore. Raw CSV files are pre-processed by a dedicated data pipeline (`data_cleaning.py`) into a cleaned, merged CSV format optimized for time-series analysis.
- APIs, Services, Integrations:
  - `/api/countries`, `/api/dates`: GET endpoints for fetching filter metadata.
  - `/api/dashboard`: POST endpoint that accepts filtering parameters and dynamically computes KPIs, trend lines, map coordinates, correlation matrices, and linear regression forecasts.
- Data Flow: Raw datasets -> Python ETL script (melt, merge, diff) -> Cleaned CSV -> FastAPI memory -> API JSON Response -> Next.js Axios Client -> Plotly UI Rendering.

3. TECHNOLOGY STACK
- Frontend: Next.js, React, Tailwind CSS, Plotly.js, Axios. Chosen for SEO/SSR capabilities, rapid UI development, and enterprise-grade, highly interactive charting.
- Backend: Python, FastAPI, Pydantic, Uvicorn. Chosen for incredibly fast API performance, automatic OpenAPI documentation, and seamless integration with Python's data science ecosystem.
- Data Science & ML: Pandas, NumPy, Scikit-learn. Chosen for vectorized data manipulation, matrix operations, and efficient linear regression modeling.
- Prototyping: Streamlit. Chosen for rapid iteration of data visualizations and model testing before committing to the Next.js production UI.
- DevOps: Docker, Docker Compose. Chosen to ensure consistent, isolated environments across development and deployment.

4. FEATURES ANALYSIS
- Major Features: Global & regional KPI tracking, daily case trend lines, top affected countries breakdown, geographic choropleth mapping, and mortality scatter plots.
- Hidden/Advanced Features: 
  - AI Forecast: An on-the-fly Scikit-learn linear regression model that predicts the next 30 days of cases for a selected country based on historical data.
  - Correlation Heatmap: Dynamically computes a correlation matrix of daily cases between different countries.
  - Data Pipeline: Automatically converts cumulative wide-format data into normalized daily differentials.
- Feature Ranking: 1. AI Forecasting, 2. Interactive Geographic Map, 3. Complex ETL Data Pipeline, 4. Dynamic Dashboard Filtering.
- Feature Complexity: The dynamic forecasting feature requires parsing time-series data into continuous numerical arrays, fitting a model, and extrapolating future dates in real-time within the API request lifecycle.

5. ENGINEERING CHALLENGES
- Technical Challenges: Handling cumulative anomalies in the raw datasets (e.g., negative daily cases due to data corrections) and transforming wide-format date columns into a long-format analytical schema.
- Implementation Details: Solved data anomalies by implementing vectorized Pandas `.diff()` operations followed by `.clip(lower=0)` to ensure data integrity for daily counts.
- Architectural Decisions: Opted to load the entire cleaned dataset into FastAPI memory during the `@app.on_event("startup")` hook. This sacrifices some RAM but achieves ultra-low latency for API responses compared to reading from disk or querying a database on every request.
- Tradeoffs: Built a file-based datastore rather than implementing PostgreSQL. This simplifies deployment and is suitable for static analytical data, but would require migrating to a real database if live, continuous data ingestion were needed.

6. DATABASE ANALYSIS
- Design Patterns: Flat-file Data Warehouse pattern utilizing pre-processed CSVs.
- Relationships: The data pipeline performs a complex relational merge of "Confirmed" and "Deaths" datasets utilizing a composite key: `['Province/State', 'Country/Region', 'Lat', 'Long', 'Date']`.
- Normalization: Applied "Tidy Data" principles by using Pandas `melt()` to unpivot date columns, transforming the dataset from wide to long format (1st Normal Form).
- Data Integrity: Handled missing spatial data (`fillna('Unknown')`) and sanitized daily differentials to prevent negative case counts.

7. SECURITY ANALYSIS
- Authentication/Authorization: Open access, designed as a public information portal.
- Validation: Utilizes FastAPI's Pydantic `BaseModel` (`DashboardRequest`) to enforce strict type checking and validation on incoming JSON payloads.
- Security Best Practices: Implemented `CORSMiddleware` in FastAPI to manage cross-origin requests securely.

8. PERFORMANCE ANALYSIS
- Optimization: The frontend utilizes Next.js dynamic imports (`next/dynamic`) with `ssr: false` for `react-plotly.js` to prevent heavy charting libraries from blocking server-side rendering and reducing the initial bundle size.
- Efficient Algorithms: The backend utilizes vectorized Pandas operations instead of iterative loops, ensuring that aggregations and correlation matrix calculations over hundreds of thousands of rows happen in milliseconds.
- Scalability: The stateless architecture of the FastAPI backend and Next.js frontend, combined with Docker Compose, allows the services to be scaled horizontally with ease.

- Complete Structure: A monolithic, server-side rendered web application following a simplified MVC architecture using Flask.
- Frontend Architecture: Utilizes Jinja2 templating engine to render server-side HTML. Bootstrap 5 is used for responsive UI components, with minimal vanilla JavaScript (`main.js`) for client-side interactions and custom CSS (`style.css`) for layout tweaks.
- Backend Architecture: Built with Python and Flask. A single `app.py` acts as the controller, handling routing, session-based authentication, and business logic (like transaction handling for billing).
- Database Architecture: Uses SQLite as a lightweight, file-based relational database. Interacts with the database via Flask-SQLAlchemy (ORM).
- APIs/Services: No external APIs; all data processing and rendering happens internally within the Flask routes.
- Data Flow: The user submits forms (e.g., generating a bill) -> Flask route captures the POST request -> SQLAlchemy validates stock and executes database transactions -> Flask renders a Jinja template (e.g., `invoice.html`) with the updated data.

3. TECHNOLOGY STACK
- Python & Flask: Chosen for rapid development, readable syntax, and suitability for small-to-medium web applications or college projects.
- Flask-SQLAlchemy: Selected to map Python classes to database tables, avoiding raw SQL queries and making database interactions safer and more Pythonic.
- SQLite: Chosen for its zero-configuration, serverless nature, making local development and deployment trivial.
- Bootstrap 5: Selected to quickly build a modern, responsive, and mobile-friendly UI without deep CSS expertise.
- HTML5/CSS3/JavaScript: Standard stack for rendering web pages.

4. FEATURES ANALYSIS
- Major Features:
  - Admin Dashboard with high-level statistics (total sales, low stock alerts).
  - Medicine Inventory Management (CRUD operations, expiry tracking).
  - Billing Engine with multi-item support and automatic invoice generation.
  - Customer Directory Management.
  - Sales History Tracking.
- Hidden/Advanced Features: Transaction rollback mechanism in billing—if a requested medicine quantity exceeds available stock, the entire bill is aborted to prevent partial/invalid transactions.
- Feature Ranking by Importance: 1. Billing Engine, 2. Inventory Management, 3. Dashboard Analytics, 4. Customer Management, 5. Authentication.
- Feature Complexity: The Billing Engine is the most complex, requiring data synchronization across four models (Medicine, Customer, Bill, Sale) in a single transaction.

5. ENGINEERING CHALLENGES
- Complex Transaction Management: Implementing the billing route required looping through multiple items, checking individual stock levels, updating inventory, and creating related `Bill` and `Sale` records. Ensuring that an error in one item rolls back the entire transaction was a key challenge solved using `db.session.rollback()`.
- Data Consistency: Maintaining accurate `Medicine.quantity` while recording `Sale.quantity_sold` to ensure inventory perfectly reflects sales.
- State Management: Implementing secure route protection using a custom `@login_required` decorator to restrict access based on Flask sessions.
- Architectural Tradeoffs: The decision to use plain-text passwords and SQLite was a tradeoff prioritizing rapid development and simplicity for a "beginner project" over production-grade security and scalability.

6. DATABASE ANALYSIS
- Design Patterns: Relational database design strictly adhering to 3rd Normal Form (3NF).
- Relationships:
  - One-to-Many: `Customer` -> `Bill` (A customer has many bills).
  - One-to-Many: `Bill` -> `Sale` (A bill comprises many line items/sales).
  - One-to-Many: `Medicine` -> `Sale` (A medicine can appear in many sales).
- Normalization: The `Sale` table effectively acts as a line-item junction table, preventing data duplication that would occur if medicines were stored directly on the `Bill` record.
- Data Integrity: Maintained through standard relational constraints (Foreign Keys, `nullable=False`, `unique=True` on admin usernames) and ORM-level transaction rollbacks.

7. SECURITY ANALYSIS
- Authentication: Session-based login system for the administrator.
- Authorization: A `@login_required` decorator intercepts unauthorized requests and redirects them to the login page.
- Validation: Basic HTML form validation combined with backend validation to prevent selling more items than are in stock.
- Security Weaknesses: The application stores administrative passwords in plain text. For production, password hashing (e.g., using `werkzeug.security.generate_password_hash`) would be mandatory.

8. PERFORMANCE ANALYSIS
- Optimization Techniques: Using SQLAlchemy relationships (`backref`, `lazy=True`) ensures queries are relatively optimized and simple to write. 
- Efficient Algorithms: Calculates dashboard statistics directly via SQL aggregate functions (e.g., `Medicine.query.count()`) rather than iterating through lists in Python memory.
- Scalability Considerations: Designed as a monolithic app with SQLite, its scalability is limited horizontally. However, the use of SQLAlchemy means migrating to a robust RDBMS like PostgreSQL for vertical/horizontal scaling is a one-line config change.

2. TECHNICAL ARCHITECTURE
================================================================================
Complete Project Structure:
A monolithic repository containing both a Vite-powered React frontend (`/src`) and an Express.js REST API backend (`/backend`).

Frontend Architecture:
- Framework: React 19 built with Vite.
- State Management: React Context API (`AppContext`) used to manage global state (cart, products, user sessions) avoiding heavy external libraries like Redux for a leaner bundle.
- Routing: Client-side routing via `react-router-dom` mapping URLs to page components (`/pages`).
- Component Design: Highly modular. Shared components (`/components` like `Navbar`, `CartDrawer`, `ProductCard`) are decoupled from page logic. Custom hooks (`useScrollReveal`) handle UI animations.

Backend Architecture:
- Framework: Node.js with Express.js.
- Architecture Pattern: MVC-inspired structure. Separated into `routes` (API endpoints), `controllers` (business logic), `models` (database schemas), and `middleware` (auth, error handling, rate limiting).
- Static File Serving: Handles user-uploaded images natively via Express static serving (`/uploads`).

Database Architecture:
- System: SQLite managed via Sequelize ORM.
- Relational Structure: Strongly typed relational schema linking `Users`, `Products`, `Categories`, `Collections`, `Orders`, and `OrderItems`.
- Associations: Complex relationships mapped out (e.g., Many-to-Many between Collections and Products, One-to-Many for User to Orders).

Data Flow:
1. Client interacts with UI (e.g., adds item to cart).
2. React Context updates local state immediately for optimistic UI rendering.
3. Axios makes async REST API calls to the Express backend.
4. Express routes request through security middleware (Rate Limiter, Auth).
5. Controller processes request, utilizing Sequelize ORM to read/write to SQLite.
6. Backend responds with JSON payload, React updates view based on server truth.

3. TECHNOLOGY STACK
================================================================================
Frontend:
- React 19: Chosen for its concurrent rendering and robust ecosystem.
- Vite: Replaced Webpack for significantly faster HMR and build times.
- Tailwind CSS v4: Chosen for utility-first, rapid, and responsive styling directly within JSX.
- Lucide React: Lightweight, consistent iconography.
- React Router DOM v7: For declarative routing.

Backend:
- Node.js & Express.js: Chosen for their asynchronous, event-driven nature, ideal for handling concurrent I/O requests in an API.
- Sequelize ORM: Chosen to abstract SQL queries, providing security against SQL injection and easy model validation.
- SQLite3: Chosen for lightweight, zero-configuration database needs, perfect for MVP scalability and easy deployments.
- JWT (JSON Web Tokens): Chosen for stateless, scalable user authentication.
- Bcryptjs: For secure, salted password hashing.
- Multer: For handling multipart/form-data (image uploads).

Interactions:
The frontend and backend communicate exclusively via a RESTful JSON API. Axios is used on the client to send HTTP requests to the Express endpoints, carrying JWTs in headers for authenticated routes.

Architectural Decisions & Tradeoffs:
- SQLite vs. PostgreSQL: Chose SQLite for rapid development, lower hosting costs, and zero-config deployment. Tradeoff: May require migration to PostgreSQL if concurrent write loads become extremely high.
- Context API vs. Redux: Chose Context API for state management to keep the architecture simple and reduce boilerplate. Tradeoff: Context can cause unnecessary re-renders if not carefully memoized, handled here by splitting contexts or strict memoization.

6. DATABASE ANALYSIS
================================================================================
Patterns: Relational Schema with ORM abstraction.
Relationships:
- 1:N (Category -> Products)
- M:N (Collections <-> Products via `CollectionProducts` junction table)
- 1:N (User -> Orders)
- 1:N (Order -> OrderItems)
- 1:N (Product -> OrderItems)
Normalization: Designed to 3rd Normal Form (3NF). Product details are separated from Order items to preserve historical pricing at the time of purchase.
Data Integrity: Enforced at the database level via Sequelize constraints (e.g., `allowNull: false`, `unique: true` for emails) and lifecycle hooks.

7. SECURITY ANALYSIS
================================================================================
- Authentication: Stateless JWT implementation. Tokens issued on login and required in `Authorization` headers for protected routes.
- Authorization: Role-based access control (RBAC) inferred from the `role` field (default: 'user') in the User model, restricting admin endpoints.
- Data Validation: Payload validation on routes using `express-validator`.
- Payload Limits: Implemented `express.json({ limit: '10mb' })` to prevent DoS attacks via massive payloads.
- Rate Limiting: Global `express-rate-limit` middleware applied to `/api` routes to throttle excessive requests.
- Password Security: `bcryptjs` hooks used at the ORM layer to automatically salt and hash passwords before insertion or update.

8. PERFORMANCE ANALYSIS

- Backend Optimizations: API responses are kept lean. SQLite in-memory caching mechanisms are utilized by Sequelize.
- Scalability: The stateless Express + JWT architecture allows the backend nodes to be horizontally scaled behind a load balancer easily, provided the SQLite DB is migrated to a centralized RDBMS like Postgres.
